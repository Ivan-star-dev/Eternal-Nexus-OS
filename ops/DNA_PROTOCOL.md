## PARTE 1 — IDENTIDADE

### O que é o Eternal Nexus OS

O Eternal Nexus OS é um sistema operacional vivo para criação, governança e evolução de um organismo digital soberano.

Não é um projeto. Não é uma equipa. É um sistema com:

- **Identidade própria** — nome, voz, visão, território
- **Pioneiros** — executores com papéis, pilares e competências distintas
- **Protocolos** — regras que eliminam ambiguidade e preservam intenção soberana
- **Ciclos** — fases de bulking, lapidação e consolidação em escada progressiva

### Natureza fractal

O sistema é fractal: cada parte contém a lógica do todo.

Um handoff bem emitido replica a estrutura do DNA.
Uma task elegível no BASTION replica a estrutura do sistema.
Um pioneiro bem calibrado replica o propósito do owner.

## PARTE 2 — REGRAS PRIMITIVAS

As Regras Primitivas são as leis absolutas do sistema.
Não têm exceção. Não têm workaround. Violá-las é deriva.

```
REGRA-1: BASTION é a fonte única de execução elegível.
         Pioneiro que não encontra task elegível no BASTION não executa.

REGRA-2: Owner é o único soberano.
         Só o owner abre gates, aprova fases, altera prioridades no BASTION.

REGRA-3: Evidência é obrigatória.
         Todo handoff inclui EVIDENCE_BLOCK com ALTERACAO_REAL explícito.

REGRA-4: Handoff é contrato.
         Pioneiro que emite handoff fecha a sessão com prova real.
         Pioneiro que recebe handoff tem contexto completo.

REGRA-5: Deriva começa onde o BASTION termina.
         Task inventada fora do BASTION é deriva. Deriva interrompe o sistema.

REGRA-6: Gate não aberto = não executável.
         Task com `PODE_ENTRAR_SOZINHO: não` — obrigatoriamente aguarda gate do owner. Sem exceção.

REGRA-7: Artefacto canônico prevalece sobre inferência.
         Em caso de conflito: docs/NEXUS_OS.md > ops/NLF.md > ops/BASTION.md > tudo.

REGRA-8: Preservar é mais importante que melhorar.
         Aceitar → Preservar → Melhorar → Lapidar. Nesta ordem. Sempre.
```

### Como os sistemas se encadeiam

```
Owner abre gate
    ↓
BASTION marca task elegível
    ↓
Pioneiro lê LIVE_STATE + BASTION (via FOL)
    ↓
IGNITION mantém fluxo contínuo
    ↓
AUTOFLOW guia a seleção da próxima task
    ↓
Pioneiro executa + emite HANDOFF_TABLE + EVIDENCE_BLOCK + CANALIZACAO_TABLE
    ↓
HANDOFF_LEDGER regista (append-only)
LIVE_STATE actualiza
BASTION actualiza
    ↓
Próxima task elegível → loop
```

## PARTE 6 — OPERAÇÕES

### Protocolo de sessão (resumo canônico)

**Ao iniciar:**
```
1. Ler LIVE_STATE seção 1 → estado atual
2. Verificar fila própria na seção 2 → gate aberto?
3. Ler semáforo na seção 3 → natureza e executor corretos?
4. Ler bloqueios na seção 4 → algum me afeta?
5. Ler últimas 2 entradas do HANDOFF_LEDGER → contexto recente?
```

**Ao encerrar:**
```
1. Actualizar cabeçalho do LIVE_STATE
2. Actualizar seção 1 (estado atual)
3. Actualizar fila própria na seção 2
4. Actualizar semáforo e linha temporal
5. Adicionar/remover bloqueios se necessário
6. Adicionar entrada no topo do HANDOFF_LEDGER
7. Emitir handoff no formato canônico
8. Commit + push
```

### Ordem de emissão de handoff (obrigatória)

```
1. HANDOFF_TABLE     — o que foi feito
2. EVIDENCE_BLOCK    — prova real
3. CANALIZACAO_TABLE — canalização atual
```

## PARTE 8 — ESPAÇO CREATOR / BASTION

### O espaço do Owner (Creator Space)

O owner é soberano absoluto. Só o owner pode:

- Abrir ou fechar gates
- Aprovar novas fases e camadas
- Alterar prioridades no BASTION
- Redirecionar pioneiros entre territórios
- Interromper ou pausar a ignição
- Definir visão, identidade, naming e direção do produto

O owner **não precisa de** justificar gates. Um gate aberto é uma ordem. Um gate fechado é uma trava.

### O BASTION como extensão do owner

O BASTION não substitui o owner — é a expressão operacional da sua intenção soberana.
Quando o owner não está presente, o BASTION fala por ele dentro dos gates já abertos.

```
Owner presente → decide ao vivo
Owner ausente  → BASTION governa com os gates que o owner deixou abertos
Conflito       → owner prevalece sempre
```

### Gates de soberania

| Gate | Significado |
|---|---|
| `PODE_ENTRAR_SOZINHO: sim` | Pioneiro entra sem confirmar com owner |
| `PODE_ENTRAR_SOZINHO: não` | Owner deve abrir explicitamente |
| `GATE: fechado` | Nenhum pioneiro actua — aguarda |
| `GATE: aberto` | Pioneiro elegível pode entrar agora |
| `GATE: aguarda-owner` | Gate em análise — não executar |
