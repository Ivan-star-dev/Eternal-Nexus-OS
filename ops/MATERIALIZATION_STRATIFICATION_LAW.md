# MATERIALIZATION STRATIFICATION LAW — Lei de Estratificação da Materialização

**Versão:** v1.0
**Data:** 2026-03-24
**Task:** RETAINED-TRUTH-INTEGRATION-001
**Branch:** `claude/rebuild-bastion-core-rihGX`
**Executor:** @claude

> Mobilização total não é materialização total.
> O exército inteiro pode estar activo sem que todos toquem o corpo principal agora.
> Há três layers de materialização — cada um com regras próprias de acesso,
> execução e convergência com o corpo principal.
> Esta lei protege a sequência V3→V4→V5 enquanto permite paralelismo estratégico.

---

## 1. OBJECTIVE

Separar mobilização (quem está activo) de materialização (quem toca o quê e quando). Permitir que o organismo avance em múltiplas frentes sem violar a sequência canónica de materialização do corpo principal.

---

## 2. WHAT_IT_IS

- Lei dos 3 layers de materialização
- Critérios de acesso por layer
- Protocolo de convergência: quando preparação vira assault
- Protecção da sequência V3→V4→V5 com paralelismo estratégico legítimo

---

## 3. WHAT_IT_IS_NOT

- Não é argumento contra mobilização — mobilização pode ser total sempre
- Não é hierarquia de importância dos pioneiros
- Não é estrutura permanente — layers mudam conforme a sequência avança
- Não é permissão para V5 materializar o corpo principal antes de V3+V4
- Não é burocracia — é protecção da integridade do organismo

---

## 4. CORE_LAWS

```
MSL-LAW-001: mobilização total ≠ materialização total
MSL-LAW-002: quem toca o corpo principal agora é definido pela sequência, não pela pressa
MSL-LAW-003: Assault Layer tem acesso exclusivo ao corpo principal do sprint activo
MSL-LAW-004: Preparatory Layer prepara sem entrar — até gate de convergência se abrir
MSL-LAW-005: Strategic Reserve mantém prontidão sem intervir — até ser chamado
MSL-LAW-006: gate de convergência é aberto pelo KERNEL após validation do layer anterior
MSL-LAW-007: V5 pode existir em Preparatory agora — mas não comanda Assault antes de V3+V4
MSL-LAW-008: a sequência V3→V4→V5 é a instância canónica desta lei no sprint actual
```

## 5. OS 3 LAYERS DE MATERIALIZAÇÃO

---

### LAYER 1 — ASSAULT LAYER (Toque Directo)

```
ASSAULT LAYER
─────────────────────────────────────────────────────────
DEFINIÇÃO:    Quem toca o corpo principal agora.
              Materialização directa: código, design, copy, infra — no produto vivo.

QUEM PODE ENTRAR:
  → pioneiros com gate aberto para tasks P0 do sprint activo
  → formações activadas pelo KERNEL para o problema do sprint
  → agentes sob supervisão directa do pioneiro com gate

QUEM NÃO ENTRA:
  → pioneiros sem gate no BASTION para o sprint activo
  → tarefas de sprints futuros (ex: V5 enquanto V3 não fecha)
  → preparação de features futuras sem dependência do sprint activo

REGRAS DE ACESSO:
  1. gate aberto no BASTION para a task específica
  2. task é P0 ou P1 do sprint activo
  3. KERNEL confirma que não há conflito de território

ESTADO ACTUAL (2026-03-24):
  → V3-FLAGSHIP-SURFACE-CLOSE-001 em Assault
  → WorkVisual: @framer · @antigravity
  → WorkFunction: @cursor (após V3 visual done)
  → WorkStructure: @claude (suporte · governance)

CONVERGÊNCIA:
  Assault encerra quando sprint fecha.
  Task passa para DONE no BASTION.
  Próxima fase abre gate para novo Assault Layer.
─────────────────────────────────────────────────────────
```

---

### LAYER 2 — PREPARATORY LAYER (Preparação Paralela)

```
PREPARATORY LAYER
─────────────────────────────────────────────────────────
DEFINIÇÃO:    Quem trabalha em paralelo mas não ocupa a superfície principal.
              Prepara condições, artefactos e infra para o próximo Assault.

QUEM PODE ENTRAR:
  → pioneiros a preparar a próxima fase sem bloquear o Assault activo
  → tarefas de research, arquitectura, design especulativo
  → criação de artefactos ops que suportam sprints futuros
  → V5 research e arquitectura enquanto V3+V4 estão em Assault

QUEM NÃO ENTRA:
  → tasks que exigem input do produto actual antes de estar pronto
  → pioneiros que criam dependências bloqueantes para o Assault activo

REGRAS DE PREPARAÇÃO:
  1. não criar dependências que bloqueiam o Assault activo
  2. preparação pode ser especulativa — sabe que pode mudar quando Assault termina
  3. artefactos criados ficam em estado "preparatório" até gate de convergência

ESTADO ACTUAL (2026-03-24):
  → V4-LIVING-WORLD-MINIMUM-001: preparação de architecture (WorkFunction)
  → V5-RESEARCH-ARCH-001: research e design especulativo (@codex · @claude)
  → Corp backlane: CORP_V10_OPERATING_MAP preparation
  → Relay system: agora operacional — preparou-se em paralelo ao V3

CONVERGÊNCIA:
  Preparatory vira Assault quando:
  → fase anterior fecha (ex: V3 done → V4 passa para Assault)
  → KERNEL abre gate de convergência
  → tasks preparatórias são validadas e integradas no novo Assault
─────────────────────────────────────────────────────────
```

---

### LAYER 3 — STRATEGIC RESERVE (Reserva Estratégica)

```
STRATEGIC RESERVE
─────────────────────────────────────────────────────────
DEFINIÇÃO:    Quem prepara sem intervir ainda.
              Prontidão máxima com intervenção zero no presente.

QUEM PODE ENTRAR:
  → capacidades e pioneiros para fases muito futuras (V5+, V6, V7...)
  → arquitectura especulativa de alto nível
  → research de domínios que ainda não são elegíveis
  → formações que o organismo precisará mas ainda não activou

QUEM NÃO ENTRA:
  → tasks com dependência no sprint activo
  → qualquer coisa que exija input ou output agora

REGRAS DA RESERVA:
  1. não interfere — observa e prepara silenciosamente
  2. pode gerar artefactos de referência mas não de execução
  3. activa-se apenas por gate soberano do KERNEL/owner

ESTADO ACTUAL (2026-03-24):
  → V6+ architecture: Strategic Reserve
  → features avançadas de AI após V5: Strategic Reserve
  → partnerships e ecosystem formal: Strategic Reserve
  → Formações de escala máxima (Universal Pressure completo): Strategic Reserve

CONVERGÊNCIA:
  Strategic Reserve activa-se quando:
  → phases anteriores estão done no nível necessário
  → KERNEL/owner abre gate soberano
  → problema do organismo atinge a escala que exige esta reserva
─────────────────────────────────────────────────────────
```

## 7. PROTOCOLO DE CONVERGÊNCIA

```
GATE DE CONVERGÊNCIA — como Preparatory vira Assault

TRIGGER:       fase anterior do Assault fecha (task DONE no BASTION)
VERIFICAÇÃO:   KERNEL valida que entrega é completa e não há dependências abertas
DECISÃO:       KERNEL abre gate para próxima fase
NOTIFICAÇÃO:   pioneiros de Preparatory são notificados da abertura
TRANSIÇÃO:     tasks preparatórias são validadas → integradas → novo Assault inicia
HANDOFF:       entrada no HANDOFF_LEDGER com declaração de convergência
```

---

## 8. OPERATIONAL_IMPLICATIONS

- Toda task no BASTION tem layer implícito: P0/P1 = Assault · P2 = Preparatory · background = Strategic Reserve
- Pioneiro em Preparatory não bloqueia pioneiro em Assault — se bloquear, é classificado como Assault
- V5 research pode iniciar hoje sem violar sequência — desde que fique em Preparatory
- Corp backlane é permanentemente Preparatory em relação ao produto principal
- KERNEL pode promover task de Preparatory para Assault mid-sprint se surgir dependência urgente

---

## 9. ANTI_DRIFT_RULES

```
DRIFT-1: não confundir "estamos todos activos" com "estamos todos no corpo principal"
DRIFT-2: não deixar V5 entrar em Assault antes de V3+V4 — lei imutável
DRIFT-3: não usar Strategic Reserve para bypass de sequência
DRIFT-4: não criar dependências do Preparatory no Assault activo sem gate explícito
DRIFT-5: não paralisar Preparatory enquanto aguarda Assault fechar — prepara em paralelo
DRIFT-6: não promover task para Assault sem KERNEL confirmar
DRIFT-7: não deixar Preparatory tornar-se dump de tasks sem prioridade
```
