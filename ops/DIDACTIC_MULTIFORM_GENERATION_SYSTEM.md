## MISSÃO

Transformar o projecto, ecossistema, produto e tecnologia em múltiplas formas de material didáctico que possa ensinar:
- pioneers
- colaboradores
- futuros operadores
- estudantes
- instituições
- futuros públicos gerais

---

## LEI CENTRAL

```
MULTIFORM GENERATION LAW
────────────────────────────────────────────────────────────
Cada sistema major deve ser traduzível em múltiplas
formas de ensino.

Uma única forma de documentação nunca chega.
O mesmo conhecimento deve existir em:
  → forma técnica
  → forma explicativa
  → forma histórica
  → forma curricular
────────────────────────────────────────────────────────────
```

---

## FRASE CANÔNICA

> Cada verdade importante do sistema deve ser ensinável em múltiplas formas,
> não apenas documentada uma vez.

---

# FAMÍLIA 1 — FORMAS DIDÁCTICAS INTERNAS

Público: pioneers, colaboradores internos, futuros operadores do sistema.

| Forma | Descrição |
|-------|-----------|
| Quick Brief | resumo executivo de 1 página — o que é, para que serve, como usar |
| Concept Guide | guia de conceito — aprofundamento de um sistema ou ideia |
| Glossary | definições canônicas de termos do ecossistema |
| System Overview | visão panorâmica de um sistema completo |
| Role Guide | como operar num papel específico (pioneer, owner, etc.) |
| Implementation Guide | como implementar / activar um protocolo ou sistema |

---

# FAMÍLIA 2 — FORMAS TÉCNICAS AVANÇADAS

Público: engenheiros, arquitectos, operators técnicos, researchers.

| Forma | Descrição |
|-------|-----------|
| Architecture Lecture | explicação formal da arquitectura de um sistema |
| Engineering Reference | referência técnica de implementação |
| Protocol Deep Dive | análise aprofundada de um protocolo canônico |
| Product Systems Dossier | dossier completo de um produto com todos os sistemas envolvidos |
| Research Notes | notas de investigação em torno de um tema técnico |
| Design Thesis | fundamento teórico de uma decisão de design ou arquitectura |

---

# FAMÍLIA 3 — FORMAS PÚBLICAS / EDUCACIONAIS

Público: estudantes, académicos, futuros cursistas, audiência pública.

| Forma | Descrição |
|-------|-----------|
| Course Outline | estrutura de curso baseada no sistema |
| Curriculum Module | módulo de currículo ensinável em contexto educacional |
| Lecture Deck Source Material | fonte para criação de apresentações de aula |
| University-Level Notes | notas de nível universitário sobre o sistema |
| Structured Explainers | explicadores estruturados de conceitos do ecossistema |
| Future Monetizable Content | material com potencial de publicação/venda como curso ou livro |

---

# FAMÍLIA 4 — FORMAS DE EVIDÊNCIA

Público: auditores, stakeholders, futuros investidores, historiadores do projecto.

| Forma | Descrição |
|-------|-----------|
| Proof of Evolution | demonstração da maturação do sistema ao longo do tempo |
| Before/After Maturity Explanation | comparação estado inicial vs. estado actual |
| Project Phase Studies | análise de uma fase do projecto — o que aconteceu, o que mudou |
| System Case Studies | caso de estudo de como um sistema resolveu um problema real |
| Design and Engineering Lessons | lições aprendidas em design e engenharia do ecossistema |

# PROTOCOLO 1 — TRADUÇÃO MULTI-FORMATO

Cada bloco major do projecto deve ser exprimível como:

```
MULTI-FORMAT TRANSLATION CHECKLIST
  □ doc de sistema canônico
  □ explainer didáctico
  □ guia técnico
  □ lição histórica
```

---

# PROTOCOLO 2 — REUTILIZAÇÃO

Material didáctico deve ser modular para poder tornar-se:

```
REUSABILITY MAP
  material interno → training interno
  training interno → bundle de documentação
  bundle de documentação → curso
  curso → produto de aprendizagem institucional
  produto institucional → conteúdo monetizável
```

**Regra:** ao criar material didáctico, pensar sempre no nível imediatamente acima de reutilização.

---

# PROTOCOLO 3 — REFRESH NA MUDANÇA DE MATURIDADE

Se o projecto matura, a camada de ensino também matura:

```
DIDACTIC REFRESH TRIGGERS
  → produto sobe de nível (V3→V5, V5→V7, etc.)
  → protocolo soberano muda de versão
  → pioneer role binding actualizado
  → nova capability fabric activada
  → novo domínio adicionado ao domain graph
  → major release de produto
```

Ao detectar qualquer trigger: marcar material existente para revisão e criar versão nova.

---

# PROTOCOLO 4 — ORGANIZAÇÃO POR SUBFAMÍLIA

Todo material deve ser organizado por:

```
TAXONOMY
  → família (internal | technical | public | evidence)
  → subfamília (concept | role | architecture | course | etc.)
  → bloco (bloco 1–7 do sistema)
  → fase (V1–V10)
  → nível de dificuldade (introductory | intermediate | advanced | expert)
  → audiência-alvo (pioneer | engineer | student | public)
```
