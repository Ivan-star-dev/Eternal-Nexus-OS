## MISSÃO

Garantir que toda decisão importante, sistema, protocolo, evolução de produto, mudança de ecossistema e virada estratégica é capturada permanentemente em:
- documentação do repositório
- materiais didácticos
- registos históricos
- sistemas de inventário
- memória viva do projecto

---

## LEI CENTRAL

```
OMNIPRESENT RECORDING LAW
────────────────────────────────────────────────────────────
Nada importante pode existir apenas em conversa.
Tudo decisivo deve tornar-se:
  1. documentação canônica
  2. explicação didáctica
  3. registo histórico
  4. referência de inventário
  5. fonte viva e actualizável
────────────────────────────────────────────────────────────
```

---

## FRASE CANÔNICA

> Nada decisivo fica só como conversa;
> tudo se torna canon, ensino, história e memória.

---

# FAMÍLIA 1 — DOCUMENTAÇÃO CANÔNICA

**O que o sistema É.**

Artefactos:
- manifestos
- protocolos
- docs de sistema
- docs de arquitectura
- docs de estratégia
- role bindings
- context pools

**Regra:** cada decisão aprovada deve ter representação canônica no repositório antes de ser considerada concluída.

---

# FAMÍLIA 2 — MATERIAL DIDÁCTICO

**Como o sistema é ENSINADO.**

Artefactos:
- docs explicativos
- walkthroughs educacionais
- guias de conceito
- material fase a fase
- glossários
- conteúdo de academia interna
- material de cursos futuros

**Regra:** cada sistema major deve ser traduzível em pelo menos uma forma de ensino antes de o sprint fechar.

---

# FAMÍLIA 3 — REGISTO HISTÓRICO

**Como o sistema é RECORDADO.**

Artefactos:
- registos de fase
- logs de decisões major
- logs de evolução
- viradas de visão
- versões anteriores
- motivo de cada mudança

**Regra:** o ecossistema não pode perder a razão pela qual algo mudou. O motivo é tão importante quanto o resultado.

---

# FAMÍLIA 4 — INVENTÁRIO E EVIDÊNCIA

**Como o sistema é AUDITÁVEL.**

Artefactos:
- inventário de assets
- inventário de projecto
- inventário de tecnologia
- inventário de conceitos
- referências de artefactos
- prova de evolução

**Regra:** qualquer auditor interno ou externo deve conseguir navegar o ecossistema sem precisar de perguntar ao owner onde as coisas estão.

---

# FAMÍLIA 5 — REFRESH E ACTUALIZAÇÃO

**Como o sistema evita obsolescência.**

Artefactos:
- actualizações de versão
- docs refrescados
- registos supersedidos
- material didáctico reescrito
- verdades antigas migradas para estado canônico novo

**Regra:** sempre que o projecto evolui, o material existente deve ser revisto, marcado como desactualizado ou adaptado.

---

# PROTOCOLO 1 — REGISTO UNIVERSAL

Toda decisão aprovada deve ser registada em pelo menos estas 3 formas:

```
UNIVERSAL REGISTRATION CHECKLIST
  □ forma canônica (doc de protocolo/sistema/arquitectura)
  □ forma didáctica (explicação ensinável)
  □ forma histórica (log de decisão/evolução)
```

---

# PROTOCOLO 2 — PROIBIÇÃO DE CONHECIMENTO ÓRFÃO

```
NO ORPHAN KNOWLEDGE RULE
────────────────────────────────────────────────────────────
Se algo existe apenas em:
  - chat / conversa
  - rascunho / memória de sessão
  - cabeça de um só pioneer

→ é conhecimento incompleto
→ deve ser capturado antes de o sprint fechar
→ se não for capturado, é tratado como não-existente
────────────────────────────────────────────────────────────
```

---

# PROTOCOLO 3 — REFRESH VIVO

Sempre que o projecto evolui:

```
REFRESH PROTOCOL
  1. Identificar docs afectados pela mudança
  2. Marcar versões desactualizadas como SUPERSEDED
  3. Adaptar material ainda válido para nova realidade
  4. Refrescar material didáctico para reflectir estado actual
  5. Confirmar que inventário está sincronizado
```

---

# PROTOCOLO 4 — OMNIPRESENÇA

Verdades importantes do sistema devem estar presentes em:

```
OMNIPRESENCE CHECKLIST
  □ docs do repositório
  □ referências de memória / context pools
  □ referências de inventário
  □ context pools dos pioneers relevantes
```

---

# PROTOCOLO 5 — TANGIBILIDADE FUTURA

Investigação paralela em encarnações físicas, dispositivos e protótipos tangíveis do sistema deve ser documentada com o mesmo rigor que o sistema digital.

---

# ÁRBOL DE CAPTURA

```
Omnipresent Recording
├── Canonical Docs        → protocolos, arquitectura, sistema
├── Didactic Docs         → ensino, walkthroughs, glossário
├── History Logs          → decisões, mudanças, versões
├── Inventory Records     → assets, artefactos, referências
└── Refresh / Update Loops → versionar, marcar, adaptar
```

---

# CAMADAS DE CAPTURA

```
Layer 1 — DECISION       Algo é aprovado.
Layer 2 — CANONIZATION   Torna-se doc de protocolo/sistema/arquitectura.
Layer 3 — DIDACTIC       Torna-se explicável e ensinável.
Layer 4 — HISTORICAL     É registado como parte da evolução do projecto.
Layer 5 — REFRESH        É revisitado sempre que o projecto matura.
```

---

# FUNÇÕES DO SISTEMA

| Função | Descrição |
|--------|-----------|
| Record | Capturar a decisão |
| Translate | Transformar em material de aprendizagem |
| Archive | Guardar traço histórico |
| Refresh | Actualizar material antigo para nova realidade |
| Keep Omnipresent | Garantir que nada decisivo desaparece do sistema |
