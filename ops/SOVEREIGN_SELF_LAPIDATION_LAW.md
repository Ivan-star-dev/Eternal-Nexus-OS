## 6. SELF_LAPIDATION_LOOP

```
SOVEREIGN SELF-LAPIDATION LOOP
═══════════════════════════════════════════════════════════

PASSO 0 — DESTINY FIRST
  O sistema lê o destino máximo coerente.
  Não V3. Não V10. V40 como forma disciplinadora.
  O destino não é meta — é norte que organiza tudo o que se produz agora.
  "O que seria o sistema mais potente e mais coerente que posso imaginar sem delírio?"

PASSO 1 — REVERSE EMBODIMENT
  O sistema retroengenharia o destino para o presente.
  Pergunta: "qual é o maior salto realizável agora dentro da fase activa?"
  Não o que quer ser amanhã — o melhor que pode ser hoje.
  Este é o acto de trazer o máximo ao real sem delirar.

PASSO 2 — INTERNAL DIAGNOSIS
  O sistema lê as suas próprias tensões, gargalos, lacunas e colisões.
  Pergunta: "onde estou abaixo do que devia ser?"
  Lê: artefactos · routing · formações · relay · runtime · canon · identidade
  Identifica o delta entre o estado actual e o melhor embodiment possível.

PASSO 3 — SELF-LAPIDATION
  O sistema trabalha sobre as suas próprias lacunas:
  → lapida: remove o que não serve, afina o que serve
  → testa: verifica se o refinamento aumenta coerência e throughput
  → optimiza: melhora routing, clarity, fruto, anti-drift
  → automatiza: onde o padrão é sólido, reduz atrito humano
  → elimina: onde há duplicação ou ruído, corta sem hesitar

PASSO 4 — INTEGRATION CHECK
  Antes de materializar lapidação no corpo principal:
  → passa no SIGNAL_TEST da One Organism Law
  → passa no LAA test da Life Alignment Axis
  → não viola a Sequence Law
  → não dilui a Mother Law
  → tem gate aberto no BASTION
  Se qualquer check falha → lapidação vai para Preparatory / Strategic Reserve

PASSO 5 — PHASE BOUNDARY DETECTION
  O sistema monitoriza continuamente o limite da fase activa.
  Pergunta: "chegámos ao máximo de lapidação possível dentro desta fase?"
  Critérios de limite:
  → todas as tasks P0 do sprint activo fechadas
  → throughput líquido estabilizado
  → nenhum gap crítico aberto com gate disponível
  → corpo principal está no melhor estado possível para esta fase

PASSO 6 — STOP + REGISTER + WAIT
  Quando o limite de fase é atingido:
  → STOP: o loop encerra para esta fase
  → REGISTER: estado completo documentado (LIVE_STATE + HANDOFF_LEDGER)
  → HANDOFF: emissão do handoff canónico com fruto e estado
  → WAIT: o sistema aguarda o Criador
  Nenhum pioneiro retoma por iniciativa própria.
  O Criador abre o próximo gate.

═══════════════════════════════════════════════════════════

## 7. PHASE_BOUNDARY_AND_STOP_CONDITION

```
STOPPING CONDITION — critérios de paragem soberana
─────────────────────────────────────────────────────────
O loop para quando TODOS os seguintes são verdade:

CRITÉRIO 1 — Tasks P0 fechadas
  todas as tasks P0 da fase activa têm status DONE no BASTION

CRITÉRIO 2 — Gaps críticos sem gate
  não há gaps críticos que tenham gate aberto na fase activa
  (gaps sem gate → próxima fase, não esta)

CRITÉRIO 3 — Throughput estabilizado
  3 ciclos consecutivos sem ganho líquido mensurável
  = o sistema atingiu o tecto desta fase

CRITÉRIO 4 — Fruto registado
  toda lapidação materializada está no HANDOFF_LEDGER com evidência real
  (fruto não registado = fruto não existente)

CRITÉRIO 5 — Corpo principal estável
  o produto está no estado mais forte possível para esta fase
  sem regressão em qualidade, identidade ou coerência

CRITÉRIO 6 — Handoff emitido
  LIVE_STATE actualizado · HANDOFF_LEDGER actualizado · artefactos commitados e pushados

QUANDO TODOS OS 6 CRITÉRIOS SÃO TRUE:
  → STOP activo
  → aguarda Criador
  → nenhum pioneiro retoma sem gate soberano
─────────────────────────────────────────────────────────

## 8. RELATION_TO_SWMR

O SWMR é simultaneamente sujeito e objecto da auto-lapidação:

- **Sujeito**: o SWMR é o mecanismo que opera o loop de lapidação em cada sessão
- **Objecto**: o SWMR é ele próprio refinado pelo loop — routing melhor, perfis mais ricos, faces mais precisas

A auto-lapidação do SWMR acontece em Preparatory Layer — não interrompe o runtime activo. A lapidação de sessão só é materializada no runtime após validation e gate.

```
SWMR + SSL LOOP
→ sessão activa: SWMR opera → gera fruto → detecta tensão
→ após sessão: SSL LOOP lê tensão → diagnóstica → lapida SWMR
→ lapidação validada → integrada no SWMR da próxima sessão
→ runtime melhora continuamente sem interromper sessão activa
```
