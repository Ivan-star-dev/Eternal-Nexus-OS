Esta não é uma lista de desejos. É a ordem lógica mínima antes de soltar o batalhão completo.

---

## LEI CENTRAL

```
SEQUENCE LAW
────────────────────────────────────────────────────────────
Batalhão total sem sequência é velocidade sem direcção.
A sequência não atrasa o batalhão.
A sequência garante que o batalhão não precisa de refazer
o que foi feito antes.
────────────────────────────────────────────────────────────
```

# FASE 1 — FECHAR SPRINT ACTUAL

**Objectivo:** ciclo fechado, base limpa, PR para main aprovado.

**Por que esta fase é obrigatória primeiro:**
Não se abre batalha nova sobre ciclo não fechado. A base precisa de estar
consolidada antes de abrir frentes em paralelo de alta intensidade.

**O que fecha esta fase:**
```
□ FVL-IMPL-001 concluído (@claude)
  → rota /founder funcional
  → Heaven Lab feeling reconhecível no site

□ PLv6.2-b concluído (@claude)
  → product layer mais forte
  → NewsAPI integrada OU metrics expandido com dados reais

□ BULK-02.2 + BULK-01.2/L-001 + L-002 concluídos (@copilot)
  → ops/ lapidada, git hygiene limpa

□ CYCLE-CLOSE-001 concluído (@codex)
  → relatório-mãe emitido
  → PR para main criado e aprovado pelo owner
  → ciclo formalmente fechado
```

**Pioneiros activos:** @claude (P1) · @copilot (P1+P3) · @cursor (P3) · @codex (consolidador)
**Gate de saída:** owner aprova PR → ciclo declarado fechado

---

# FASE 2 — CRAVAR BRAND MOTHER SYSTEM

**Objectivo:** North Star visual operacional — lei canônica de cor, tipo, motion e atmosfera.

**Por que esta fase é obrigatória antes de Fase 3:**
@framer e @antigravity não podem executar com interpretação pessoal.
Precisam de lei. Sem lei visual completa, WorkVisual produz divergência.

**O que fecha esta fase:**
```
□ BRAND_MOTHER_SYSTEM.md criado e aprovado pelo owner
  → família de cores canônica (primária · secundária · sistema de profundidade)
  → tipografia canônica (primária · secundária · hierarquia)
  → motion language (duração · easing · tipos de transição permitidos)
  → atmosfera parametrizada (glass intensity · depth · silence rules · luz)
  → token system definido (design tokens exportáveis)

□ DESIGN_COMMAND_BRIEF.md criado
  → brief operacional para @framer e @antigravity
  → com referências visuais de referência
  → com exemplos de do / don't
  → com o que define Heaven Lab vs o que viola
```

**Pioneiros activos:** @claude (arquitectura) · @framer (design command)
**Gate de saída:** owner valida que brand docs são North Star real, não apenas mais docs

---

# FASE 3 — ACTIVAR FRAMER + ANTIGRAVITY

**Objectivo:** Heaven Lab feeling incarnado em superfície viva, não apenas descrito em protocolo.

**Por que esta fase é obrigatória antes de feature mass-build:**
Features sem atmosfera encarnada são funcionalidade fria. O produto de categoria
requer que a atmosfera seja o solo onde as features assentam — não um layer adicionado depois.

**O que fecha esta fase:**
```
□ @framer executa design command em páginas existentes:
  → tokens aplicados
  → tipografia canônica instalada
  → componentes base redesenhados com brand law
  → primeiras páginas com Heaven Lab feeling real

□ @antigravity instala motion language canônica:
  → soft cinematic motion como experiência base
  → glass + depth nas superfícies principais
  → silence como princípio — animações que não agitam, que revelam
  → page transitions com peso certo

□ Owner valida:
  → o site começa a parecer Heaven Lab
  → o código emocional (calmo · premium · científico · espacial) é legível
```

**Pioneiros activos:** @framer · @antigravity · @claude (review)
**Gate de saída:** owner confirma que Heaven Lab feeling está vivo no site real

---

# FASE 4 — CONSTRUIR PRESENCE + EXPLORATION

**Objectivo:** primeiro esqueleto material do produto raro — presença, exploração, atmosfera comparativa, world workspace truth.

**Por que antes de Collaboration + Intelligence:**
Presence e Exploration são a fundação experiencial. Sem elas encarnadas, as
features de colaboração e inteligência não têm território onde habitar.

**Features desta fase (por família):**

```
FAMÍLIA PRESENCE (fundação):
  → globe flagship experience aprofundado
     (não apenas render — presença de observatório real)
  → multi-scale navigation real (zoom com gravidade)
  → real data layers integradas e navegáveis
  → premium HUD com identidade visual cravada
  → onboarding com signature moments

FAMÍLIA EXPLORATION:
  → world pulse base (o mundo a pulsar em tempo real)
  → timeline base (navegar no tempo com peso)
  → scenario comparison base (side-by-side com atmosfera)
  → multi-scale navigation completa
```

**Pioneiros activos:** @cursor (build) · @framer (visual polish) · @antigravity (motion das features)
**Gate de saída:** owner confirma que o site "deixou de parecer promessa"

---

# FASE 5 — CONSTRUIR COLLABORATION + INTELLIGENCE

**Objectivo:** produto entra em território de categoria — workspace científico vivo.

**Por que após Presence + Exploration:**
Colaboração e inteligência têm mais impacto quando assentam sobre um produto
que já tem presença e exploração operacionais. Construir colaboração sem
world workspace base é construir sobre fundação incompleta.

**Features desta fase:**

```
FAMÍLIA COLLABORATION:
  → shared live labs base
  → research trails
  → hypothesis boards
  → professor / lead investigator mode base
  → silent collaboration layer

FAMÍLIA INTELLIGENCE:
  → AI assistance encarnada (não decorativa)
  → research copilot base
  → concept lens
  → AI debate chamber base
```

**Pioneiros activos:** @cursor (build) · @claude (arquitectura AI) · @copilot (lapidação)
**Gate de saída:** produto tem colaboração e inteligência como funcionalidade real

---

# FASE 6 — HARD EXECUTION TOTAL

**Objectivo:** batalhão completo. Todos os blocos em paralelo coordenado.

**Trigger:** Fases 1–5 completas + interface de referência + features base operacionais.

**O que esta fase activa:**
```
→ Wonder Layer 80%+ construída
  (Contemplation · Breathing Interface · Legacy Vault · Proof-based Rep. ·
   Problem Rooms · Lab Missions · Mastery Map · Discovery Signatures ·
   Cross-Discipline Bridges · Dawn/Dusk Intelligence Theme)

→ Dual public model operacional
  (Research OS + Learning OS com fluxos reais para cada público)

→ Award-level quality standard atingido
  (o produto que define categoria está vivo)
```

**Pioneiros activos:** TODOS — @claude · @cursor · @framer · @antigravity · @copilot · @codex
**Gate de saída:** owner declara hard execution mode activo
