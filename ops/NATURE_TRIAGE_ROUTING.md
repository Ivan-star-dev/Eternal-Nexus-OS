# NATURE TRIAGE AND ROUTING LOGIC

**Versão:** v1.0

**Sealed by:** owner (instrução directa)

---

## Purpose

Define how incoming material must be classified before it is turned into
architecture, implementation, documentation, visual language, or response.

Goals:
- prevent context mixing
- route matter to the correct part of the organism
- preserve canonical clarity
- let the architect arrive confused while the system returns organised structure

---

## Core Rule

Layers define where something belongs.
Natures define what kind of matter it is.

Before acting on any input → classify its nature.
Wrong nature → wrong routing → implementation drift → organism weakens.

---

## Canonical Natures

### FACE
Defines: interface · visual identity · atmosphere · sensory language ·
presentation · beauty · user-facing narrative

Question: **Is this about how the organism appears, feels, or is perceived?**

### ESPINHA
Defines: architecture · runtime · topology · structural logic · layers ·
membranes · scales · internal system design

Question: **Is this about how the organism is built, connected, or sustained internally?**

### CÓDICE
Defines: canon · law · protocol · rules · standards · memory ·
governance · decision logic

Question: **Is this about how the organism thinks, decides, governs, or remembers?**

### FORJA
Defines: implementation · code · CI/CD · rollout · migrations ·
integration · task execution · materialisation

Question: **Is this about building, wiring, shipping, or operationalising?**

### ALARME
Defines: risk · incident · failure · urgency · containment ·
active breakage · emergency response

Question: **Is this about immediate threat, instability, blockage, or protection under pressure?**

---

## Triage Sequence

For every meaningful input, identify:

1. Primary nature
2. Secondary natures if present
3. Safe or unsafe mix
4. Canonical layer
5. Target output form
6. Whether it is: vision · blueprint · implementation · audit ·
   correction · expansion · emergency · memory/update

---

## Safe Mixing Law

### Same-nature (always allowed)
FACE+FACE · ESPINHA+ESPINHA · CÓDICE+CÓDICE · FORJA+FORJA · ALARME+ALARME

### Conditional (allowed when structurally useful)
ESPINHA+FORJA · CÓDICE+ESPINHA · CÓDICE+FORJA
FACE+FORJA (when visual implementation is already intended)
ALARME+any (when urgency directly affects that layer)

### Unsafe (must be separated first)
FACE+ALARME (unless true visual crisis)
FACE+CÓDICE (unless formalising frozen visual law)
raw ESPINHA+FACE (without stable system spine)
FORJA+CÓDICE (when doctrine and execution still unresolved)

## Routing Logic

| Nature  | Routes Into |
|---------|-------------|
| FACE    | Product · Marketing/Narrative · Book visual system · Founder public surfaces |
| ESPINHA | System Spine · Architecture docs · Membranes · Scales · Runtime models |
| CÓDICE  | Sovereign Core · Governance · Canon · Laws · Standards · Protocol files |
| FORJA   | Forja · Codebase · CI/CD · Rollout · Handoffs · Mechanical tasks |
| ALARME  | Mother Shield · Audit systems · Incident models · Risk correction · Active recovery |

## Output Forms

| Nature  | Canonical Output |
|---------|-----------------|
| FACE    | Visual direction block · book spread brief · UI direction |
| ESPINHA | Architecture block · topology doc · membrane diagram |
| CÓDICE  | Canonical law block · protocol file · governance doc |
| FORJA   | Implementation handoff · task force packet · commit |
| ALARME  | Emergency action packet · audit panel · incident report |

Do not force one output form on all natures.

```

```

Mandatory when ambiguity is high.

## Failure Conditions

| Wrong routing | Consequence |
|---------------|-------------|
| FACE treated as ESPINHA | organism becomes pretty but hollow |
| ESPINHA treated as FACE | architecture turns decorative |
| CÓDICE treated as FORJA too early | doctrine mutated by execution pressure |
| FORJA without ESPINHA | implementation without spine |
| ALARME ignored | system breaks silently |
