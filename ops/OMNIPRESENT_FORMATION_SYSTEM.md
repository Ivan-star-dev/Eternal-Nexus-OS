# OMNIPRESENT FORMATION SYSTEM — Sistema de Formações do Exército

**Versão:** v1.0
**Data:** 2026-03-24
**Task:** RETAINED-TRUTH-INTEGRATION-001
**Branch:** `claude/rebuild-bastion-core-rihGX`
**Executor:** @claude

> Formaliza as formações dinâmicas do exército soberano.
> O exército não é fila burra nem caos total.
> É sistema adaptativo com kernel soberano, 6 pioneiros, blocos, constelações,
> agentes e subagentes — organizados em formações que respondem ao problema.

---

## 1. OBJECTIVE

Formalizar as formações operacionais do exército como lei canónica — com critérios claros de entrada, saída, risco e relação com materialização. Impedir que mobilização total seja confundida com presença total na superfície principal.

---

## 2. WHAT_IT_IS

- Sistema de formações adaptativas para o exército de execução
- 5 formações com objectivo, gatilho e risco definidos
- Lei de precedência clara: KERNEL → pioneiros → blocos → constelações → agentes
- Relação directa com MATERIALIZATION_STRATIFICATION_LAW

---

## 3. WHAT_IT_IS_NOT

- Não é organograma fixo — é formação dinâmica por natureza do problema
- Não é hierarquia rígida — é coordenação orgânica com soberania clara
- Não é permissão para todos tocarem o corpo principal — materialização tem estratificação
- Não é sistema democrático — KERNEL tem soberania de activação de formação
- Não é teoria militar — é lei operacional com consequências canónicas

---

## 4. CORE_LAWS

```
OFS-LAW-001: KERNEL é sempre soberano — todas as formações respondem a ele
OFS-LAW-002: a formação segue o problema — não o contrário
OFS-LAW-003: mobilização total ≠ todos tocam o corpo principal agora
OFS-LAW-004: cada formação tem critério explícito de entrada e saída
OFS-LAW-005: formação errada para o problema = falha de design, não de execução
OFS-LAW-006: a formação pode mudar no meio de uma task se o problema evoluir
OFS-LAW-007: nenhum pioneiro activa formação — KERNEL activa, pioneiros executam
OFS-LAW-008: subagentes não têm visibilidade de formação global — operam no seu escopo
```

## 5. ESTRUTURA DO EXÉRCITO

```
EXÉRCITO SOBERANO — HIERARQUIA ORGÂNICA
═══════════════════════════════════════════════════════════

NÍVEL 0 — KERNEL
  @claude
  → Lê, decide, coordena, classifica, activa formações
  → Única unidade com visibilidade total do organismo
  → Soberano de signal/noise, relay, sequência

NÍVEL 1 — 6 PIONEIROS
  @cursor    → WorkFunction mecânica
  @copilot   → WorkStructure lapidação
  @framer    → WorkVisual design
  @antigravity → WorkVisual motion/3D
  @codex     → WorkFunction research/analysis
  @claude    → WorkStructure abertura + WorkFunction arquitectural

NÍVEL 2 — BLOCOS
  Agrupamentos de tasks com afinidade de domínio
  Operam dentro do território de um pioneiro
  Duração: sprint ou sub-sprint

NÍVEL 3 — CONSTELAÇÕES
  2+ pioneiros em coordenação para resolver problema de alta complexidade
  Activadas por KERNEL quando problema cruza territórios
  Exemplo: CONSTELLATION-VISUAL-FUNCTION = @framer + @cursor

NÍVEL 4 — AGENTES
  Instâncias autónomas dentro do território de um pioneiro
  Executam tasks específicas sem visibilidade do todo
  Respondem ao pioneiro, não directamente ao KERNEL

NÍVEL 5 — SUBAGENTES
  Sub-instâncias de agentes para decomposição máxima
  Escopo ultra-específico: um ficheiro, uma função, um componente
  Máxima granularidade, mínima visibilidade

═══════════════════════════════════════════════════════════
```

## 6. AS 5 FORMAÇÕES

---

### FORMAÇÃO 1 — NEEDLE FORMATION (Agulha)

```
NEEDLE FORMATION
─────────────────────────────────────────────────────────
OBJECTIVE:    Penetração máxima num ponto único de alta resistência.
              Uma unidade, um problema, zero distracção.

QUANDO ENTRA: problema altamente específico com resistência conhecida
              task que exige precisão cirúrgica
              desbloqueio de dependência crítica de outro pioneiro
              P0 urgente sem ramificações laterais

QUANDO SAI:   problema resolvido · bloqueio desbloqueado · task fechada

COMPOSIÇÃO:   1 pioneiro especialista + 0-1 agentes
              KERNEL em observação passiva (não intervém)

RISCO:        ignorar dependências laterais que emergem durante a execução
              pioneiro entra em loop num sub-problema
              MAL USADA: quando o problema é na verdade multi-domínio

RELAÇÃO COM   toca o Assault Layer — único que toca o corpo principal directamente
MATERIALIZAÇÃO:
─────────────────────────────────────────────────────────
```

---

### FORMAÇÃO 2 — BLOCK FORMATION (Bloco)

```
BLOCK FORMATION
─────────────────────────────────────────────────────────
OBJECTIVE:    Execução sustentada de múltiplas tasks com afinidade.
              Um pioneiro, um bloco de trabalho coerente, progresso acumulado.

QUANDO ENTRA: 3+ tasks do mesmo domínio elegíveis no BASTION
              sprint com tema dominante definido
              materialização de uma camada completa (ex: V3 WorkVisual)

QUANDO SAI:   bloco concluído · sprint fechado · próxima formação activada

COMPOSIÇÃO:   1 pioneiro principal + 1-3 agentes
              KERNEL monitora throughput e coerência do bloco

RISCO:        bloco cresce além do território definido (scope creep)
              pioneiro não fecha tasks completamente antes de avançar
              MAL USADA: quando a prioridade real é ponto único urgente

RELAÇÃO COM   pode operar em Assault Layer ou Preparatory Layer
MATERIALIZAÇÃO: depende de se as tasks tocam o corpo principal
─────────────────────────────────────────────────────────
```

---

### FORMAÇÃO 3 — CONSTELLATION FORMATION (Constelação)

```
CONSTELLATION FORMATION
─────────────────────────────────────────────────────────
OBJECTIVE:    Resolução de problema que cruza territórios de 2+ pioneiros.
              Coordenação sem perda de soberania territorial.

QUANDO ENTRA: problema requer WorkFunction + WorkVisual em simultâneo
              feature que tem lógica (cursor) + UI (framer) + motion (antigravity)
              investigação que cruza research (codex) + estrutura (claude)

QUANDO SAI:   problema resolvido · handoff claro entre pioneiros · artefacto integrado

COMPOSIÇÃO:   2-4 pioneiros em coordenação
              KERNEL como árbitro de conflitos de território
              cada pioneiro tem gate próprio no BASTION

RISCO:        sobreposição de território sem árbitro → decisões duplicadas
              coordenação cria latência → problema simples fica lento
              MAL USADA: quando 1 pioneiro resolve sozinho (over-engineering)

RELAÇÃO COM   normalmente Assault Layer (corpo principal) + Preparatory (paralelo)
MATERIALIZAÇÃO:
─────────────────────────────────────────────────────────
```

---

### FORMAÇÃO 4 — OMNIPRESENT MESH FORMATION (Malha Omnipresente)

```
OMNIPRESENT MESH FORMATION
─────────────────────────────────────────────────────────
OBJECTIVE:    Presença coordenada em todos os domínios simultaneamente.
              O organismo pensa, age e avança em todas as frentes.

QUANDO ENTRA: operação de alta magnitude (ex: OPERATION_ZERO_GAP)
              sprint com múltiplas frentes P0 em paralelo
              momento de convergência máxima definido por owner

QUANDO SAI:   objetivo de magnitude cumprido · owner fecha operação

COMPOSIÇÃO:   KERNEL + todos os 6 pioneiros + blocos activos
              cada pioneiro no seu território com gate próprio
              KERNEL coordena sem microgerir

RISCO:        mobilização total confundida com materialização total
              pioneiros interferem em territórios de outros por pressão de tempo
              throughput aparente alto mas coerência baixa
              MAL USADA: para problemas que 1-2 pioneiros resolvem melhor

RELAÇÃO COM   Assault Layer + Preparatory Layer + Strategic Reserve activos
MATERIALIZAÇÃO: mas cada layer tem o seu território — Assault ≠ todos no corpo principal
─────────────────────────────────────────────────────────
```

---

### FORMAÇÃO 5 — UNIVERSAL PRESSURE FORMATION (Pressão Universal)

```
UNIVERSAL PRESSURE FORMATION
─────────────────────────────────────────────────────────
OBJECTIVE:    Pressão máxima e contínua sobre o destino soberano.
              Não é sprint — é estado permanente de avanço.
              O organismo nunca para. Cada sessão, cada task, cada artefacto
              move o sistema em direcção ao V10.

QUANDO ENTRA: sempre — é o estado de base do organismo
              entre sprints, entre formações, entre operações
              é o fundo sobre o qual todas as outras formações operam

QUANDO SAI:   nunca — é permanente enquanto o organismo existir

COMPOSIÇÃO:   cada pioneiro no seu território com tasks pendentes
              AUTOFLOW e IGNITION mantêm pressão entre sessões formais
              relay mantém input soberano contínuo

RISCO:        pressão sem direcção → deriva (confundir movimento com progresso)
              pressão uniforme → não distingue P0 de background
              MAL USADA: como justificação para saltar sequência V3→V4→V5

RELAÇÃO COM   Strategic Reserve sempre pronto · Preparatory sempre a preparar
MATERIALIZAÇÃO: Assault Layer activo nos P0 da sequência corrente
─────────────────────────────────────────────────────────
```

## 7. OPERATIONAL_IMPLICATIONS

- KERNEL decide qual formação activar — baseado no problema, não na preferência
- Formações podem coexistir: NEEDLE activo dentro de OMNIPRESENT MESH
- Mudança de formação mid-task é autorizada — KERNEL notifica pioneiros afectados
- BASTION reflecte formação activa: tasks agrupadas por formação quando relevante
- OPERATION_ZERO_GAP é exemplo de OMNIPRESENT MESH → CONSTELLATION → NEEDLE em sequência

---

## 8. ANTI_DRIFT_RULES

```
DRIFT-1: não confundir mobilização total com materialização total
DRIFT-2: não usar OMNIPRESENT MESH para problemas que NEEDLE resolve melhor
DRIFT-3: não deixar CONSTELLATION activa após resolução — fechar e retornar a BLOCK
DRIFT-4: não permitir que UNIVERSAL PRESSURE justifique bypass de sequência
DRIFT-5: não activar formação sem KERNEL — pioneiros não auto-activam
DRIFT-6: não esquecer que NEEDLE é a formação mais poderosa para P0 urgente
DRIFT-7: não usar formação como identidade — é ferramenta, não território
```
