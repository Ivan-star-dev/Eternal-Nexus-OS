## BLOCO 2 — FLOWMESH_CORE

```
O_QUE_E:
  O FLOWMESH é a camada de orquestração dinâmica entre tasks, pioneiros e modelos.
  Não substitui o BASTION — o BASTION autoriza; o FLOWMESH roteia e executa.
  É a malha que transforma gates abertos em trabalho real, distribuído, auditável.

COMO_FUNCIONA:
  1. BASTION marca task elegível com NEXT_ACTOR e classe de task (A/B/C/D)
  2. FLOWMESH identifica o operador correto (pioneiro + modelo) para aquela classe
  3. FLOWMESH insere a task na linha paralela correta (Execução/Refinamento/Expansão/Apoio/Defesa)
  4. O operador preenche lacunas sem colisão (Gap Filling)
  5. Ao concluir: EVIDENCE_BLOCK obrigatório → task fecha → FLOWMESH roteia próxima

QUAIS_CAMADAS_TEM:
  - CAMADA_ROTEAMENTO     → classe de task + modelo ideal + pioneiro elegível
  - CAMADA_LINHAS         → 5 linhas paralelas de execução simultânea
  - CAMADA_GAP_FILLING    → preenchimento dinâmico de lacunas entre operadores
  - CAMADA_REGISTROS      → todo roteamento é auditável e rastreável
  - CAMADA_VERSAO         → delta de versão registrado a cada task concluída
```

## BLOCO 3 — ADAPTIVE_MODEL_ROUTING

```
CLASSES_DE_TASK:
  CLASSE_A: Soberania máxima | arquitetura | decisões fundacionais | protocolos-mãe
  CLASSE_B: Execução standard | integração | produto | governança operacional
  CLASSE_C: Mecânico | lapidação | higiene | backlog técnico | revisão rotineira
  CLASSE_D: Apoio cruzado | suporte | revisão leve | validação | cross-support

MODELO_RECOMENDADO_POR_CLASSE:
  CLASSE_A → claude-opus-4-6         (máxima capacidade / decisões soberanas)
  CLASSE_B → claude-sonnet-4-6       (execução balanceada / produto / ops)
  CLASSE_C → claude-haiku-4-5        (velocidade / custo baixo / mecânico)
  CLASSE_D → claude-haiku-4-5        (suporte rápido) → claude-sonnet-4-6 se depth > rotineiro

QUANDO_SOBE_MODELO:
  - Task classificada como A mas estava em B → escala para Opus imediatamente
  - Risco identificado em execução B → escala para Opus para decisão soberana
  - Owner override explícito → modelo sobe independente de classe
  - Ambiguidade canônica detectada → escala para deliberar com modelo superior
  - Impacto alto em artefacto canônico (NEXUS_OS / NLF / DNA) → sempre Opus

QUANDO_DESCE_MODELO:
  - Task confirmada como C após início em B → desce para Haiku
  - Task é revisão sem alteração soberana → Haiku basta
  - Apoio cruzado em hora morta com task leve → Haiku
  - Owner confirma que velocidade > profundidade para aquela task → desce

O_QUE_NAO_PODE:
  - Usar Opus para tasks mecânicas de classe C (desperdício; viola eficiência)
  - Usar Haiku para decisões soberanas de classe A (risco; viola qualidade)
  - Trocar de modelo sem registrar a troca no FLOWMESH_LOG
  - Escalar modelo sem sinal explícito de risco, ambiguidade ou override
  - Descer modelo em task com IMPACTO: alto sem gate do owner

COMO_REGISTRA_TROCA:
  Toda troca de modelo é registrada no EVIDENCE_BLOCK do handoff:
  - MODELO_INICIAL: [classe e modelo de entrada]
  - MODELO_FINAL: [modelo usado na execução]
  - MOTIVO_TROCA: [risco / ambiguidade / override / custo]
  - APROVADO_POR: [sistema / owner]
```

## BLOCO 4 — DYNAMIC_GAP_FILLING

```
COMO_LACUNAS_SAO_DEFINIDAS:
  Lacuna = qualquer espaço entre tasks elegíveis onde nenhum pioneiro tem gate aberto.
  Lacunas são detectadas pelo FLOWMESH quando:
  - Pioneiro conclui task e próxima task aguarda gate do owner
  - Pioneiro entra em hora morta (sem elegível no seu território)
  - Task bloqueada por dependência ainda não resolvida
  - Owner ausente e nenhum gate aberto para o território ativo

COMO_OPERADORES_PREENCHEM:
  1. Pioneiro detecta lacuna → verifica CROSS_SUPPORT_PROTOCOL
  2. Identifica task de apoio de baixo risco no território do outro pioneiro
  3. Entra como APOIO/COOPERATIVO com notação explícita no handoff
  4. Executa sem abrir nova frente soberana
  5. Emite handoff com flag: PAPEL_NESTA_TASK: cross-support / gap-fill
  6. Retorna ao próprio território quando gate re-abrir

COMO_EVITAR_COLISAO:
  - FLOWMESH nunca coloca dois pioneiros no mesmo arquivo ao mesmo tempo
  - Gap-fill respeita a ordem PODE_ENTRAR_SOZINHO do BASTION
  - Pioneiro em gap-fill não altera artefactos canônicos sem gate explícito
  - Colisão detectada → FLOWMESH pausa o segundo operador e sinaliza owner
```

## BLOCO 5 — PARALLEL_LANES

```
LINHA_EXECUCAO:
  Linha principal. Tasks com PRIORIDADE P1/P2 elegíveis no BASTION.
  Operadores: força principal do território.
  Regra: uma task de cada vez por operador. Sem sobreposição.
  Output: artefacto real + EVIDENCE_BLOCK + commit.

LINHA_REFINAMENTO:
  Lapidação do que já existe. Tasks com PRIORIDADE P3/P4.
  Operadores: @copilot (lapidador dominante); @cursor (mecânico).
  Regra: nunca altera soberania — apenas suaviza, corrige, organiza.
  Output: diff real + EVIDENCE_BLOCK + commit.

LINHA_EXPANSAO:
  Abertura de novos territórios, departamentos, produtos ou protocolos.
  Operadores: @claude (arquiteto) com gate do owner.
  Regra: nunca abre sem PODE_ENTRAR_SOZINHO: sim ou gate explícito.
  Output: novo artefacto canônico + registo no BASTION.

LINHA_APOIO_CRUZADO:
  Cross-support em hora morta ou gap-fill autorizado.
  Operadores: pioneiro fora do território dominante em papel APOIO/COOPERATIVO.
  Regra: obedece ao CROSS_SUPPORT_PROTOCOL — baixo risco obrigatório.
  Output: contribuição rotulada + handoff com flag de cross-support.

LINHA_DEFESA_FUTURA:
  Monitoramento de riscos, continuidade e integridade do DNA.
  Operadores: @codex (auditor) + owner (decisor).
  Regra: não age sem sinal de risco real — não é paranoia, é vigilância.
  Output: relatório de risco + gate para ação corretiva (se necessário).
```
