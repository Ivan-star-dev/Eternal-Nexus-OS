## PARTE 3 — NEURAL SIGNALS

Os sinais são os eventos que propagam intenção e mudança de estado pela rede.

```
NEURAL_SIGNAL_SYSTEM ═════════════════════════════════════════════════════════
SINAL            │ ORIGEM          │ DESTINO         │ EFEITO
─────────────────┼─────────────────┼─────────────────┼────────────────────────
TASK_SIGNAL      │ BASTION         │ PIONEER_GRID    │ task elegível ativada
GATE_SIGNAL      │ owner           │ BASTION         │ gate aberto/fechado
PRIORITY_SIGNAL  │ owner/BASTION   │ PIONEER_GRID    │ P1→P4 define urgência
RETURN_SIGNAL    │ CROSS_SUPPORT   │ PIONEER_GRID    │ pioneiro retorna ao território
SUPPORT_SIGNAL   │ PIONEER_GRID    │ CROSS_SUPPORT   │ apoio cruzado ativado
VERSION_SIGNAL   │ qualquer nó     │ HANDOFF_LEDGER  │ delta de versão registrado
HANDOFF_SIGNAL   │ PIONEER_GRID    │ OUTPUT_LAYER    │ sessão encerrada com evidência
ROUTE_SIGNAL     │ FLOWMESH        │ PIONEER_GRID    │ modelo correto atribuído
IGNITION_SIGNAL  │ IGNITION        │ todos os nós    │ fluxo contínuo mantido
SOVEREIGN_SIGNAL │ owner           │ todos os nós    │ override soberano — prevalece sempre
══════════════════════════════════════════════════════════════════════════════
```

### Propagação de sinais

```
GATE_SIGNAL (owner abre gate)
    → TASK_SIGNAL emitido pelo BASTION
    → ROUTE_SIGNAL emitido pelo FLOWMESH (modelo atribuído)
    → PRIORITY_SIGNAL propaga urgência
    → PIONEER_GRID recebe task + modelo + prioridade
    → execução inicia
    → HANDOFF_SIGNAL emitido ao concluir
    → VERSION_SIGNAL registado no HANDOFF_LEDGER
    → IGNITION_SIGNAL mantém loop ativo
```

## PARTE 6 — NEURAL BEHAVIOR PROTOCOL

Como o sistema se comporta em cada estado operacional.

```
NEURAL_BEHAVIOR_PROTOCOL ═════════════════════════════════════════════════════

ESTADO: IGNIÇÃO_ATIVA
  → BASTION governa fila
  → pioneiros lêem BASTION antes de cada sessão
  → FLOWMESH atribui modelo correto
  → AUTOFLOW guia seleção da próxima task
  → owner só entra para gate, visão, trava ou redirecionamento

ESTADO: HORA_MORTA (sem task elegível)
  → CROSS_SUPPORT_PROTOCOL ativa
  → pioneiro em hora morta mapeia lacunas
  → apoio somente a tarefas de risco baixo/médio
  → retorno ao território ao primeiro sinal de task elegível

ESTADO: BLOQUEIO_REAL
  → execução para
  → bloqueio registado em LIVE_STATE seção 4
  → owner notificado no semáforo
  → pioneiro aguarda gate soberano sem inventar alternativa

ESTADO: GATE_ABERTO (nova task elegível)
  → FLOWMESH roteia modelo
  → PIONEER_GRID recebe task + modelo + prioridade
  → EVIDENCE_BLOCK obrigatório ao concluir
  → HANDOFF_TABLE + CANALIZACAO_TABLE emitidos
  → LIVE_STATE + BASTION + HANDOFF_LEDGER actualizados

ESTADO: DERIVA_DETECTADA
  → execução para imediatamente
  → pioneiro regista no handoff: "deriva detectada — aguardando gate"
  → owner decide se continua, redireciona ou descarta
  → REGRA-5 do DNA prevalece: "deriva começa onde o BASTION termina"
══════════════════════════════════════════════════════════════════════════════
```
