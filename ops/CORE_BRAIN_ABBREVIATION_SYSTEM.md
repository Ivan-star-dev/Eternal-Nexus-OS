# O NÚCLEO — 6 SIGLAS

## PCSE — Produto e ecossistema crescem em sequência clonada

| Campo | Conteúdo |
|-------|----------|
| Sigla | PCSE |
| Nome completo | Protocol Cloned Sequence Evolution |
| Ficheiro | `ops/PROTOCOL_CLONED_SEQUENCE_EVOLUTION.md` |
| Família | Núcleo de crescimento |
| Bloco | Bloco 2 — Ecossistema e Corporação |

**Lei comprimida:**
> O produto não cresce sozinho. O ecossistema também não cresce sozinho.
> Toda melhoria relevante deve perguntar: como isto melhora o produto?
> E como isto melhora a máquina que constrói o produto?

**Resultado:** crescimento em sequência clonada — não em trilhas separadas.

**Check operacional:**
```
PCSE CHECK
  → estamos a crescer produto + ecossistema ao mesmo tempo?
  → a melhoria de ecossistema reflecte-se em capacidade de produto?
  → a melhoria de produto eleva os sistemas que a suportam?
```

---

## ESER — Autoconstrução com auto-teste em 10 dimensões

| Campo | Conteúdo |
|-------|----------|
| Sigla | ESER |
| Nome completo | Ecosystem Self-Evolution Routine |
| Ficheiro | `ops/ECOSYSTEM_SELF_EVOLUTION_ROUTINE.md` |
| Família | Núcleo de autoevolução |
| Bloco | Bloco 6 — Execução e Governança |

**Lei comprimida:**
> O ecossistema precisa de se testar, se revisar, se corrigir,
> se actualizar e se elevar. O auto-teste não pode ser simbólico.

**10 dimensões de auto-teste:**

```
01  identidade        → o sistema continua alinhado com o DNA?
02  produto           → o produto está a maturar correctamente?
03  dev excellence    → a qualidade técnica está a subir?
04  routing           → o roteamento de tasks está correcto?
05  protocolo         → os protocolos estão actualizados?
06  memória           → nada importante foi perdido?
07  documentação      → docs estão sincronizados com a realidade?
08  feature maturity  → features atingiram o nível certo de V10?
09  toolchain/stack   → ferramentas e stack estão optimizados?
10  V10 alignment     → a trajectória aponta para V10?
```

**Check operacional:**
```
ESER CHECK
  → auto-teste está a ocorrer em dimensões reais, não simbólicas?
  → as 10 dimensões foram verificadas no sprint actual?
  → há plano de correcção para cada dimensão com gap?
```

## DMGS — Ensino multiforme disponível

| Campo | Conteúdo |
|-------|----------|
| Sigla | DMGS |
| Nome completo | Didactic Multiform Generation System |
| Ficheiro | `ops/DIDACTIC_MULTIFORM_GENERATION_SYSTEM.md` |
| Família | Núcleo de ensino |
| Bloco | Bloco 7 — Documentação e Memória |

**Lei comprimida:**
> Tudo importante precisa poder ser ensinado.
> Cada sistema relevante deve poder virar explicação, guia técnico,
> módulo educacional e prova de evolução.

**4 famílias de output:**

```
01  Internal Didactic Forms    → quick brief, concept guide, glossary, role guide
02  Advanced Technical Forms   → architecture lecture, engineering reference, protocol deep dive
03  Public/Educational Forms   → course outline, curriculum module, structured explainers
04  Evidence Forms             → proof of evolution, case studies, design lessons
```

**Resultado:** o ecossistema não só produz — também gera capital didáctico.

**Check operacional:**
```
DMGS CHECK
  → este sistema tem pelo menos uma forma de ensino criada ou planeada?
  → o material existente está actualizado com a maturidade actual?
  → há reusability path definido para este material?
```

## PPLBT — Cérebro autónomo de triagem e roteamento

| Campo | Conteúdo |
|-------|----------|
| Sigla | PPLBT |
| Nome completo | Protocol Problem Launch and Brain Test |
| Ficheiro | `ops/PROTOCOL_PROBLEM_LAUNCH_AND_BRAIN_TEST.md` |
| Família | Núcleo de cérebro autónomo |
| Bloco | Bloco 6 — Execução e Governança |

**Lei comprimida:**
> O utilizador lança o problema.
> A máquina escolhe o caminho.

**Fluxo comprimido:**
```
INPUT   → problema bruto (owner não escolhe nada)
          ↓
TRIAGE  → domínio · escala · bloco · urgência · ownership · V10
          ↓
OUTPUT  → classified as · routed to · mode · branch · closure target
```

**A task activa — não o chat:**
```
triagem → ownership → rota → workspace → branch → formação
```

**Check operacional:**
```
PPLBT CHECK
  → o problema foi roteado pelo sistema, não pelo owner manualmente?
  → a triagem operou nas 6 dimensões?
  → o routing output foi produzido em formato canônico?
  → brain test success conditions foram verificadas?
```
