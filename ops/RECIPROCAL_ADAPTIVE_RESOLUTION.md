> O problema não é objecto passivo esperando resolução.
> É campo activo que reconfigura o enxame enquanto é resolvido.
> A resolução nasce da adaptação recíproca: o sistema adapta-se ao problema
> e o problema reorganiza a formação do sistema.
> Double adaptation — lei canónica.

---

## 1. OBJECTIVE

Formalizar que a resolução de qualquer problema no organismo é um processo de adaptação mútua — não de aplicação unilateral de metodologia. O sistema lê o problema, adapta a formação. O problema, ao ser atacado, revela novos contornos que forçam nova adaptação. A resolução emerge desta dança.

---

## 2. WHAT_IT_IS

- Lei de double adaptation: sistema ↔ problema
- Modelo de como gaps emergem e formações mudam mid-task
- Formalização do momento de Resolution Collapse (convergência final)
- Princípio de Problem-Induced Formation Shift

---

## 3. WHAT_IT_IS_NOT

- Não é argumento para instabilidade ou indecisão
- Não é permissão para mudar de formação sem critério
- Não é relativismo operacional — a lei é canónica, a aplicação é adaptativa
- Não é equivalente a agilidade fraca (mudar porque é difícil)
- Não é substituição do plano — é reconhecimento de que o plano encontra o real

## 5. STRUCTURE — OS 4 MECANISMOS

### 5.1 PROBLEM-INDUCED FORMATION SHIFT

```
PROBLEM-INDUCED FORMATION SHIFT
─────────────────────────────────────────────────────────
DEFINIÇÃO:
  O problema, ao ser atacado, revela nova natureza ou complexidade.
  Esta revelação força o sistema a mudar de formação mid-task.

EXEMPLOS:
  → task aparece como NEEDLE (1 pioneiro) mas revela multi-domínio
    → shift para CONSTELLATION (2-3 pioneiros)
  → BLOCK_FORMATION bloqueia em dependência inesperada
    → shift para NEEDLE em sub-problema específico + retorno ao BLOCK
  → OMNIPRESENT MESH descobre que o problema central é único
    → shift para NEEDLE no P0 real + outros em PREPARATORY

COMO RECONHECER SHIFT NECESSÁRIO:
  ✓ pioneiro encontra dependência fora do seu território
  ✓ task cresce além do gate definido no BASTION
  ✓ solução válida para sub-problema quebra solução de outro sub-problema
  ✓ progresso para mas problema ainda existe em forma diferente

PROTOCOLO DE SHIFT:
  1. pioneiro comunica ao KERNEL: "problema mudou de natureza"
  2. KERNEL avalia: confirma shift ou redefine escopo
  3. se shift confirmado → nova formação activada → BASTION actualizado
  4. task original fechada ou re-aberta com nova definição
─────────────────────────────────────────────────────────
```

### 5.2 GAP EMERGENCE PRINCIPLE

```
GAP EMERGENCE PRINCIPLE
─────────────────────────────────────────────────────────
DEFINIÇÃO:
  Gaps não existem antes de o problema ser atacado.
  Eles emergem no acto de resolução — são revelados pelo contacto
  entre a formação e o campo do problema.

TIPOS DE GAPS:
  GAP-CONHECIMENTO:   o sistema não sabe o suficiente para resolver
  GAP-DEPENDÊNCIA:    resolução requer artefacto/decisão não existente
  GAP-TERRITÓRIO:     problema cruza fronteira de dois pioneiros sem árbitro
  GAP-SEQUÊNCIA:      resolução exige passo que está bloqueado pela sequência V3→V4→V5
  GAP-RETENÇÃO:       verdade necessária existe mas não está cravada no sistema

COMO TRATAR GAPS:
  → GAP-CONHECIMENTO    → CODEX investiga · KERNEL valida
  → GAP-DEPENDÊNCIA     → KERNEL abre gate para task de desbloqueio em BASTION
  → GAP-TERRITÓRIO      → KERNEL activa árbitro · define quem tem soberania
  → GAP-SEQUÊNCIA       → KERNEL comunica ao owner · task entra em modo DEFER
  → GAP-RETENÇÃO        → KERNEL cravar no CASCADE_RETENTION_CANON · relay se nasceu de ChatGPT

PRINCÍPIO:
  Gap identificado é gap resolvível.
  Gap ignorado é falha silenciosa.
  O sistema que nomeia gaps com honestidade é mais forte que o que os esconde.
─────────────────────────────────────────────────────────
```

### 5.3 RECIPROCAL ADAPTIVE RESOLUTION

```
RECIPROCAL ADAPTIVE RESOLUTION — O Mecanismo Central
─────────────────────────────────────────────────────────
DEFINIÇÃO:
  Resolução real não é aplicação de plano a problema.
  É diálogo activo entre formação e campo do problema.

  SISTEMA → lê o problema → adapta formação
  PROBLEMA → revela nova natureza → força nova adaptação do sistema
  SISTEMA → adapta novamente → resolve sub-conjunto
  PROBLEMA → diminui · ou · revela novo contorno
  [ciclo até Resolution Collapse]

TRÊS ESTADOS NO CICLO:
  1. READING:      sistema lê o problema · identifica natureza e escala
  2. ADAPTING:     sistema ajusta formação · KERNEL decide · pioneiros reposicionam
  3. RESOLVING:    formação encontra o campo · progresso real acontece

PRINCÍPIO DA RECIPROCIDADE:
  O sistema que não muda enquanto o problema muda está a combater o fantasma
  da versão inicial do problema — não o problema real.

  O problema que não muda conforme é atacado não é problema real —
  é tarefa mecânica. Tarefas mecânicas não precisam de adaptação recíproca.
─────────────────────────────────────────────────────────
```

---

### 5.4 RESOLUTION COLLAPSE

```
RESOLUTION COLLAPSE — O Momento de Convergência
─────────────────────────────────────────────────────────
DEFINIÇÃO:
  Momento em que o ciclo de double adaptation converge.
  O problema deixa de gerar novos contornos.
  A formação encontrou a configuração que resolve.
  A resolução colapsa de processo para resultado.

COMO RECONHECER:
  ✓ problema não gera novos gaps quando atacado
  ✓ todos os sub-problemas identificados têm solução clara ou gate definido
  ✓ formação pode fechar sem deixar dependências abertas
  ✓ artefacto/output produzido é coerente com o DNA do organismo
  ✓ KERNEL confirma: "este problema está resolvido no escopo definido"

TIPOS DE COLLAPSE:
  CLEAN COLLAPSE:       problema resolvido completamente · task fecha
  PARTIAL COLLAPSE:     sub-conjunto resolvido · rest vai para BASTION como nova task
  DEFERRED COLLAPSE:    resolução válida mas bloqueada por sequência → DEFER com gate claro

ANTI-PADRÕES:
  ✗ fechar task antes do collapse (falsa resolução)
  ✗ nunca deixar o ciclo convergir (loop de adaptação infinita)
  ✗ confundir collapse parcial com clean collapse
─────────────────────────────────────────────────────────
```

## 6. OPERATIONAL_IMPLICATIONS

- Pioneiros reportam mudança de natureza do problema — não tentam resolver sozinhos o que mudou de escopo
- KERNEL lê relatórios de mudança antes de decidir formation shift
- BASTION é actualizado quando formation shift acontece — tasks reflectem nova realidade
- Resolution Collapse é critério de fecho de task — não tempo decorrido nem esforço aplicado
- Gap Emergence alimenta CREATOR_RELAY_INBOX se o gap veio de diálogo externo

## 7. ANTI_DRIFT_RULES

```
DRIFT-1: não usar double adaptation como desculpa para instabilidade crónica
DRIFT-2: não fazer formation shift sem comunicar ao KERNEL — autonomia tem limites
DRIFT-3: não confundir GAP-SEQUÊNCIA com GAP-DEPENDÊNCIA — têm tratamentos diferentes
DRIFT-4: não fechar task em partial collapse sem criar nova task para o restante
DRIFT-5: não deixar ciclo de adaptação correr > 3 iterações sem resolution — sinalizar ao KERNEL
DRIFT-6: não usar "problema mudou" para fugir de task difícil — KERNEL avalia honestidade
DRIFT-7: não ignorar gap identificado por outro pioneiro — gaps são informação do organismo
```
