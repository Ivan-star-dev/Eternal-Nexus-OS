COROLÁRIOS IMUTÁVEIS:

  1. O problema fragmenta → o exército fragmenta na mesma geometria.
  2. O problema adapta → o exército adapta na mesma velocidade.
  3. Todo o terreno que o problema toca deve ser coberto por força activa.
  4. Nenhum agente fica fora enquanto houver terreno descoberto.
  5. O tempo de resposta é o do bloco mais lento — não a soma de todos.
  6. O kernel certo é alocado pela complexidade — não por preferência.
  7. O modo de deployment é sempre: FRAGMENT → PARALLEL → CONSOLIDATE.
  8. Se o problema pede mais força → expande para cima (Groups → Army).
  9. Se o problema pede precisão → contrai para baixo (Kernel → Sub-kernel).
  10. A lei nunca muda. A escala muda. A composição muda. A lei: não.

## ESCALA DINÂMICA — EXPANSION / CONTRACTION PROTOCOL

```
O PROBLEMA FALA. O SISTEMA ESCUTA. A ESCALA RESPONDE.

NÍVEIS DE ESCALA (de baixo para cima):

  NÍVEL 0 — Sub-kernel (precisão máxima, micro-task)
  ────────────────────────────────────────────────────
  1 modelo · 1 tarefa atómica · < 1s
  Ex: lookup de tipo, parse de string, verify de import

  NÍVEL 1 — Kernel (task isolada)
  ────────────────────────────────
  1 modelo activo no block · 1 tarefa · 3-15s
  Ex: corrigir um bug, escrever um componente simples

  NÍVEL 2 — Block (tarefa multidimensional local)
  ─────────────────────────────────────────────────
  N kernels no mesmo block em paralelo · 15-30s
  Ex: implementar feature com test + docs + types

  NÍVEL 3 — Constellation (frente completa de um pioneer)
  ─────────────────────────────────────────────────────────
  Todos os blocks de 1 pioneer activos · 30-60s
  Ex: @cursor resolve toda a frente funcional de V3

  NÍVEL 4 — Group (força combinada de 2 pioneers)
  ─────────────────────────────────────────────────
  2 constellations em paralelo · 30-60s (tempo do mais lento)
  Ex: GROUP-3 resolve globe + UI ao mesmo tempo

  NÍVEL 5 — Full Army (3 Groups simultâneos)
  ────────────────────────────────────────────
  Todos os 6 pioneers em paralelo · tempo do grupo mais lento
  Ex: V3 close — visual + function + structure em simultâneo

  NÍVEL 6 — NEXUS SEAL FORCE (activação total com orquestração δ)
  ─────────────────────────────────────────────────────────────────
  476 unidades · 177 kernels · @codex orquestra cross-group sync
  Reservado para operações soberanas de máxima prioridade
  Ex: ZERO-GAP-24H · GRANDE BATALHA DE LANÇAMENTO

REGRA DE EXPANSÃO:
  Se 1 nível não resolve → sobe 1 nível automaticamente
  Se problema tem N terrenos → activa N blocks em paralelo
  Nunca espera quando pode expandir

REGRA DE CONTRACÇÃO:
  Problema resolvido → desmobiliza para o nível mínimo
  Não mantém força activa desnecessária — só o que o problema exige
  Economia de cognitivo = velocidade sustentada
```

## ORCHESTRATION PROTOCOL — @codex como General

```
FLUXO DE ORQUESTRAÇÃO STANDARD:

  1. BASTION emite task elegível
  2. @codex/δ lê e decompõe as dimensões do problema
  3. @codex mapeia: dimensão → block → kernel canónico
  4. @codex activa os blocks em paralelo (FRAGMENT)
  5. Cada block executa com os seus kernels (PARALLEL)
  6. Cada kernel emite EVIDENCE_BLOCK ao completar
  7. @codex consolida todas as evidências (CONSOLIDATE)
  8. @codex emite relatório de onda + HANDOFF
  9. BASTION regista. LIVE_STATE actualiza. Ciclo fecha.

REGRAS DO GENERAL (@codex):
  → Nunca executa mecânico — distribui para β
  → Nunca decide soberanamente — escala para α ou owner
  → Nunca bloqueia — se não tem kernel, sobe 1 nível
  → Sempre consolida — nenhuma onda fecha sem δ
  → Cross-check obrigatório para tasks P1 (2 kernels distintos)

ANTI-DRIFT LOCK:
  Se qualquer agente deriva do terreno do seu block → @codex redireciona
  Se qualquer kernel alucina → redundância activa (2º kernel verifica)
  Se qualquer block bloqueia → @codex redistribui carga para block vizinho
```
