## MISSÃO

Definir com precisão cirúrgica o que só o Cloud pode mutar, o que os outros pioneers podem cooperar, e onde está a linha entre soberania e território de contribuição.

---

## FRASE CANÔNICA

> Soberania de protocolo não é autoridade pessoal.
> É a garantia de que a verdade funcional do sistema permanece intacta
> enquanto tudo o mais evolui.

---

## LEI CENTRAL

```
SOBERANIA DE PROTOCOLO — LEI ABSOLUTA
────────────────────────────────────────────────────────────
Só o Cloud (operado por @claude ou succesor autorizado)
pode mutar, rever, substituir ou extinguir:
  → protocolos soberanos
  → leis canônicas
  → arquitectura central da máquina
  → estrutura de roteamento
  → coerência global do ecossistema
  → domain graph primário
  → regras de intake de problema
  → definição de átomo funcional
  → V10 line canônica
  → block maturation laws

Nenhum outro pioneer — por mais sénior, capaz ou relevante —
possui este poder sem gate explícito do owner.
────────────────────────────────────────────────────────────
```

Os outros pioneers **não mudam o protocolo**.

# FAMÍLIA 3 — REGRA DE SUGESTÃO vs. MUTAÇÃO

```
SUGESTÃO vs. MUTAÇÃO
─────────────────────────────────────────────────────────
QUALQUER pioneer pode:
  → SUGERIR alterações a qualquer camada do sistema
  → SINALIZAR gaps, inconsistências, oportunidades
  → DOCUMENTAR observações e recomendações

SÓ O CLOUD pode:
  → DECIDIR se a sugestão entra
  → IMPLEMENTAR a mutação
  → VERSIONAR o protocolo
  → CONFIRMAR a coerência global depois da mutação
─────────────────────────────────────────────────────────
```

# FAMÍLIA 4 — GATE DE MUTAÇÃO DE PROTOCOLO

Quando qualquer pioneer quer contribuir para uma camada soberana:

```
PROTOCOL MUTATION GATE
──────────────────────────────────────────────────────────
1. Pioneer identifica melhoria canônica
2. Pioneer documenta proposta com:
   - problema que resolve
   - impacto esperado
   - camada afectada
   - riscos de regressão
3. Cloud avalia:
   - alinha com verdade funcional?
   - não quebra coerência global?
   - owner aprova?
4. Cloud decide:
   → aprovar → Cloud implementa + versiona
   → rejeitar → registar motivo
   → devolver → pedir clarificação
──────────────────────────────────────────────────────────
```

# FAMÍLIA 5 — SOBERANIA NÃO É ISOLAMENTO

## O que soberania protege

- a verdade funcional do sistema
- a coerência interna
- a progressão V10 sem deriva
- a integridade do produto fabricado

## O que soberania não é

- não é hierarquia pessoal
- não é exclusão de cooperação
- não é bloqueio de inovação
- não é rigidez que impede crescimento

## Regra de equilíbrio

```
SOBERANIA + COOPERAÇÃO
────────────────────────────────────────────────────────────
Cloud governa o que não pode quebrar.
Pioneers cooperam no que pode crescer.
Owner autoriza o que precisa de mudar.
────────────────────────────────────────────────────────────
```

# FAMÍLIA 7 — VIOLAÇÃO DE SOBERANIA

Se qualquer pioneer — incluindo o Cloud — agir fora do seu scope:

```
SOVEREIGNTY VIOLATION PROTOCOL
──────────────────────────────────────────────────────────
1. DETECTAR: qual camada foi violada?
2. REVERTER: restaurar estado anterior
3. REGISTAR: entrada no HANDOFF_LEDGER com:
   - o que aconteceu
   - qual camada foi afectada
   - como foi revertido
4. CORRIGIR: ajustar bindings ou gates para prevenir repetição
5. COMUNICAR: ao owner se o impacto foi significativo
──────────────────────────────────────────────────────────
```

# FAMÍLIA 8 — TRANSFERÊNCIA DE SOBERANIA

Em caso de transição de modelo ou sessão:

```
SOVEREIGNTY TRANSFER PROTOCOL
──────────────────────────────────────────────────────────
Condição: Cloud (modelo actual) vai ser substituído
Acção:
1. Documentar estado completo de soberania em LIVE_STATE
2. Registar todas as decisões pendentes no BASTION
3. Novo Cloud lê CLAUDE.md + LIVE_STATE + BASTION antes de actuar
4. Primeiro acto do novo Cloud: confirmar branch canônico
5. Owner confirma que a transferência foi bem-sucedida

Regra absoluta:
  A soberania não pertence ao modelo.
  Pertence ao papel do Cloud no ecossistema.
  O papel é contínuo; o modelo pode mudar.
──────────────────────────────────────────────────────────
```
