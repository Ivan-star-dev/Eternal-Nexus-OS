## PORTAL MICRO-LAWS

### Creation Lab
- core work bay must always be instantly legible
- project continuity surfaces must never compete with active creation space
- tool emergence must be proximity-based and context-based
- prototype pods must isolate complexity without fragmenting identity
- memory wall must reflect progress without becoming dashboard clutter
- concept surfaces must privilege structure over decoration
- entry must feel like resuming a real build session

### Learning School
- learning path must always be traversable from simple to deep
- guidance intensity must scale with maturity and confusion signals
- knowledge stations must never overload with simultaneous concepts
- demonstration spaces must show causal understanding, not static explanation
- reflection zone must reduce cognitive speed before next step
- progress feedback must increase confidence without gamified cheapness

### Simulation Hub
- every simulation surface must map to a testable model or scenario
- feedback loops must be visible but not noisy
- scenario changes must preserve orientation
- multiple variables must reveal progressively, not all at once
- system map views must be readable at macro and local level
- simulation objects must feel instrument-grade, not toy-grade

### Deep Focus Chamber
- no non-essential signals remain active
- object count must be lowest of all portals
- guidance must only appear at high necessity thresholds
- transitions in and out must protect concentration
- atmospheric motion must be minimal and slow
- focus state must feel protected, not empty

### Memory Archive / Progress Nexus
- memory must be navigable by meaning, not just chronology
- artifacts must show lineage and relationship
- milestones must feel earned and spatially embodied
- archive must support review, retrieval, reflection, and re-entry
- the space must feel alive, not like cold storage
- progress must be emotionally real but disciplined

### Collaboration Atrium
- multi-user presence must preserve low social noise
- shared surfaces must maintain authorship clarity
- collaboration views must never overwhelm solo continuity
- co-presence should feel synchronized, not crowded
- shared context must be lightweight and recoverable
- transitions between solo and collaborative modes must preserve work integrity

## SESSION MICRO-STATES

```
1.  arrival
2.  recognition
3.  stabilization
4.  continuity restore
5.  intent confirmation
6.  portal anchoring
7.  active flow
8.  guided adjustment
9.  deep work
10. portal transition
11. checkpoint save
12. closure
13. evolution prep
14. re-entry priming
```

### Session State Laws

**Arrival**
- no overload in first seconds
- continuity anchors first, complexity later

**Recognition**
- confirm identity through environment behavior, not text-heavy messaging

**Stabilization**
- restore familiarity before introducing novelty

**Continuity Restore**
- bring back exact work context before optional enhancements

**Intent Confirmation**
- infer softly through environment configuration; avoid interrogation

**Portal Anchoring**
- portal identity must lock in quickly and clearly

**Active Flow**
- remove friction and keep tool exposure contextual

**Guided Adjustment**
- assist only when a real gap appears

**Deep Work**
- lower noise, reduce change frequency, protect rhythm

**Portal Transition**
- maintain emotional thread and cognitive map

**Checkpoint Save**
- persist without interruption

**Closure**
- end gracefully, preserve momentum for next session

**Evolution Prep**
- compute refinements after value capture

**Re-entry Priming**
- next session should begin smarter, not just restored

## SPAWN LOGIC

### Spawn Priority Order
1. restore exact task context
2. restore exact portal
3. restore exact location
4. restore active surfaces
5. restore relevant artifacts
6. restore atmosphere continuity
7. re-open optional adjacent utilities only if beneficial

### Spawn Laws
- spawn must feel immediate and grounded
- user must know where they are instantly
- first interaction must be obvious
- environment should suggest continuity without shouting it
- no cold-start emptiness if a continuity record exists

---

## TRANSITION LOGIC

### Portal Transitions Must
- preserve spatial dignity
- avoid abrupt UI mode-switch feeling
- maintain cognitive map
- carry one strong continuity thread
- use architecture/light/motion to signal change
- avoid spectacle-driven transitions

### Transition Bans
- hard cuts that erase context
- sudden density spikes
- identity blending
- full resets without reason
- decorative transition effects with no orientation value

---

## OBJECT EXPOSURE RULES

- only show what is useful now
- keep adjacent utilities discoverable but dormant
- hide or soften non-essential objects in focus-heavy states
- expose high-value artifacts near likely reuse moments
- do not flood the world with "available features"
- object presence must reinforce function, not inventory

## ANTI-CHAOS LOCKS — EXPANDED

### Hard Caps
- density cap
- motion cap
- highlight cap
- simultaneous panel cap
- portal blend cap
- object category cap
- ambient signal cap
- tutorial intrusion cap

### Chaos Triggers
- more than one competing focal surface
- more than one dominant highlight cluster
- guidance appearing without context
- portal objects leaking into wrong portal identity
- visual novelty at the wrong moment
- layout mutation that breaks route memory

### When Chaos Is Detected
- reduce visible objects
- collapse secondary surfaces
- restore portal anchor
- lower motion
- simplify guidance
- prioritize continuity surfaces
- re-stabilize atmosphere
