## MISSÃO

Definir os pulsos operacionais que mantêm o ecossistema alinhado, auto-corrigido e em melhoria contínua enquanto constrói o produto.

---

## LEI CENTRAL

O ecossistema não deve agir aleatoriamente ou derivar por entusiasmo local.
Deve operar através de **pulsos:**
- pulsos de identidade
- pulsos de qualidade de desenvolvimento
- pulsos de fidelidade de produto
- pulsos de disciplina de execução
- pulsos de refinamento

Estes pulsos mantêm os pioneers próximos de:
- o projecto
- a melhor prática de desenvolvimento 2026
- o produto final pretendido

---

# FAMÍLIA 1 — IDENTITY PULSE

## Função

Perguntar continuamente:
- isto ainda parece Earth Lab?
- isto ainda pertence ao ecossistema?
- isto ainda protege o Heaven Lab feeling?
- isto ainda suporta a identidade world workspace?

## Comportamento de Pulso Preferido 2026

Identidade deve ser verificada:
- antes de construir
- durante a construção
- depois de construir

## Backup / Cooperação

Checkpoints de revisão de identidade agendados se verificação contínua ainda não estiver automatizada.

---

# FAMÍLIA 2 — DEV EXCELLENCE PULSE

## Função

Perguntar continuamente:
- estamos a programar ao standard 2026 mais forte realista?
- estamos a usar padrões de alta alavancagem?
- estamos a preservar rollback, testing, observabilidade e performance?
- estamos a depender de hábitos desatualizados?

## Comportamento de Pulso Preferido

Cada move de engenharia é verificado contra:
- version control forte · safe deployment · observabilidade
- platform leverage · AI-assisted productivity · code quality review

## Backup / Cooperação

Auditoria periódica de engenharia em todo o repo, workflow e ferramentas.

---

# FAMÍLIA 3 — PRODUCT FIDELITY PULSE

## Função

Perguntar continuamente:
- o site ainda está atrasado em relação à visão?
- o que está ainda apenas definido mas não encarnado?
- que bloco permanece sub-materializado?
- o que deve ser preenchido a seguir?

## Comportamento de Pulso Preferido

Usar o canonical checkup e a V10 line para medir:
- maturidade conceptual · maturidade de produto visível
- feature embodiment gap · atmosphere/brand gap
- research/learning gap

## Backup / Cooperação

Revisões de gap baseadas em milestones.

---

# FAMÍLIA 4 — EXECUTION PULSE

## Função

Manter o trabalho no caminho:
- branch discipline · task routing · lógica do Bastion
- alinhamento de actores · uso correcto de dispatch
- sem branch contamination

## Comportamento de Pulso Preferido

Cada task valida:
- branch actual · branch esperado
- mission fit · actor fit · closure state

## Backup / Cooperação

Triage manual e branch guard prompts explícitos.

---

# FAMÍLIA 5 — REFINEMENT PULSE

## Função

Prevenir estagnação de "good enough".

## Comportamento de Pulso Preferido

Cada deliverable é desafiado por:
- polish · flagship bar · anti-generic filter
- category-power filter · rarity filter
- performance/trust filter

## Backup / Cooperação

Passes de refinamento dedicados depois de cada milestone estrutural.

---

# FAMÍLIA 6 — CASCADE MEMORY PULSE

## Função

Prevenir esquecimento.
Garantir que novo trabalho herda:
- protocolos anteriores · decisões anteriores · sistemas anteriores
- itens aprovados não resolvidos · estado de maturidade dos blocos

## Comportamento de Pulso Preferido

Cada resposta e cada task começa de:
- cascade retention · block maturation
- V10 line · canonical delivery

## Backup / Cooperação

Docs de checkup/master revistos no início de cada ciclo.

---

# FAMÍLIA 7 — MULTI-BLOCK MATURATION PULSE

## Função

Prevenir que uma área corra à frente enquanto o resto fica imaturo.

## Comportamento de Pulso Preferido

Rastrear todos os blocos do projecto juntos:

| Bloco | Território |
|-------|-----------|
| 1 | Visão e Identidade |
| 2 | Ecossistema e Corporação |
| 3 | Produto |
| 4 | Interface / Brand / Atmosfera |
| 5 | Features e Matéria Viva |
| 6 | Execução e Governança |
| 7 | Documentação e Memória |

Depois perguntar:
- que bloco maturou? · que bloco está a passar fome?
- o que precisa de avanço sincronizado?

## Backup / Cooperação

Revisão periódica do mapa de maturidade bloco-a-bloco.

---

# FAMÍLIA 8 — AI / PIONEER ROLE PULSE

## Função

Manter os pioneers especializados mas alinhados.

## Comportamento de Pulso Preferido

Cada pioneer ou agente deve continuamente saber:
- que território é seu · que canonical pool usar
- que regras de identidade se aplicam · que barra de build-quality se aplica
- qual é o próximo target de maturidade

## Backup / Cooperação

Task briefs específicos por papel gerados a partir de docs canônicos.

---

# FAMÍLIA 9 — CURRENT-STATE / FINAL-STATE PULSE

## Função

Manter o ecossistema ancorado entre:
- onde o projecto está agora
- onde o projecto deve ir

## Comportamento de Pulso Preferido

Cada move major responde:
1. qual é o estado actual?
2. qual é o estado V10 desejado?
3. qual é o maturity gap?
4. qual é a bridge actual mais forte?

## Backup / Cooperação

Revisão formal de phase-placement antes de decisões major.

---

# FAMÍLIA 10 — CHECKLIST DE PULSOS PARA PIONEERS

Antes de trabalho major, perguntar:
1. **Identity pulse** — ainda pertence?
2. **Dev pulse** — é prática 2026 de topo?
3. **Product pulse** — encarna a visão?
4. **Execution pulse** — está no branch / mission correcto?
5. **Refinement pulse** — é raro o suficiente?
6. **Cascade pulse** — estamos a esquecer algo já aprovado?
7. **Multi-block pulse** — estamos a matar outro bloco por overfocus aqui?
