## THE SIX DOMAINS

### DOMAIN A — VALUE CREATION

**Territory:** What can be built that stores and delivers value.

**Purified into these categories:**
- Digital product systems (books, courses, info products, SaaS structure, toolkits)
- Authority systems (positioning, intellectual territory, thought leadership architecture)
- Offer systems (how value is packaged, named, bounded, and priced)
- Leverage systems (what creates 1→many from 1→1 effort)
- Brandable intellectual assets (frameworks, models, methodologies, named systems)

**Core law:** Value is durable when it solves a real problem, stores expertise structurally, and can be transmitted without the creator's presence.

**Anti-pattern:** Creating a product because it's trending, not because a proven problem demands it.

**Benchmark standard:** A value creation artifact should be able to outlive the person who made it by at least 10 years without becoming false.

---

### DOMAIN B — MARKET / DISTRIBUTION / ACQUISITION

**Territory:** How value reaches people who need it.

**Purified into:**
- Attention capture systems (hooks, category distinctness, entry point clarity)
- Audience building systems (trust accumulation, repeat engagement, community formation)
- Channel strategy (platform fit, leverage points, signal-to-noise optimization)
- Positioning logic (category ownership, contrast framing, competitor displacement)
- Trust-building mechanics (consistency signals, proof structures, authority indicators)
- Niche domination maps (who is underserved, what signal they read, how to own their attention)

**Core law:** Distribution without positioning creates noise. Positioning without distribution creates irrelevance. The combination must be engineered, not stumbled into.

**Anti-pattern:** Omnipresent content without a strategic gravity center — builds following, not authority.

**Benchmark standard:** A distribution system should create inbound pull, not only outbound push. Proof: people share it without being asked.

---

### DOMAIN C — MONETIZATION / OFFER / CONVERSION

**Territory:** How value is exchanged for resources (money, attention, time, access).

**Purified into:**
- Value ladders (sequenced offer architecture, from free to premium)
- Offer design (naming, framing, promise, proof, price anchor, risk reversal)
- Pricing logic (price as positioning signal, not just cost-recovery math)
- Package architecture (what's included, what's excluded, why)
- Upsell / ascension logic (what comes after the first exchange)
- Continuity-backed monetization (recurring value, subscription logic, retention engineering)
- Trust-based selling (buyer journey, objection maps, credibility sequences)
- Recurring monetization (systems that compound, not reset)

**Core law:** Conversion is earned by the precision of the match between problem felt and solution offered — not by persuasion mechanics.

**Anti-pattern:** Optimizing conversion rate without understanding why people don't buy (usually mismatch, not friction).

**Benchmark standard:** An offer should be buyable by a skeptical, intelligent buyer who has read everything your competitors offer. If it only converts the uncritical, it will not survive.

---

### DOMAIN D — EXECUTION / OPERATIONS

**Territory:** How value is reliably produced, repeated, and compounded.

**Purified into:**
- SOP systems (documented, transferable, version-controlled procedures)
- Workflow systems (dependency-aware, sequenced task architectures)
- Execution maps (ordered build sequences from zero to shipped)
- Operational ladders (progression from solo-operator to systems-dependent operation)
- Decision trees (if/then flows that eliminate daily judgment overhead)
- Asset pipelines (input → transformation → output chains)
- Repeatable production frameworks (templates, checklists, and protocols that preserve quality across repetitions)

**Core law:** Execution without a system is creativity. Execution with a system is leverage. Only leverage scales.

**Anti-pattern:** Heroic single-person execution that cannot be documented, transferred, or replicated.

**Benchmark standard:** Any execution system should produce the same quality output with a different operator using only the documented protocol. If it requires the original person, it's not a system — it's a skill.

---

### DOMAIN E — TEACHING / MASTERY

**Territory:** How knowledge transfers from one level of competence to a higher one.

**Purified into:**
- Curriculum architecture (what must be taught, in what order, with what dependencies)
- Learning journeys (experience design from zero to capable)
- Skill ladders (competence stages with observable proof at each level)
- Practical transformation systems (before state → after state with evidence)
- Mastery maps (what separates beginner/practitioner/expert in this domain)
- Implementation teaching (teaching that produces action, not just comprehension)
- Strategy teaching (second-order thinking, pattern recognition, framework application)
- Monetization teaching (how value creation becomes sustainable economic engine)
- Business architecture teaching (how systems connect, reinforce, and compound)
- Practical doctrine transfer (distilled laws that survive domain shifts)

**Core law:** Teaching that produces comprehension without behavioral change is entertainment. Teaching that produces permanent capability upgrade is mastery infrastructure.

**Anti-pattern:** Information overload without competency sequencing — produces informed helplessness.

**Benchmark standard:** A teaching system should produce demonstrably different behavior after completion, measurable by the learner themselves without external validation.

---

### DOMAIN F — TESTING / VALIDATION

**Territory:** How to know if a value proposition, offer, or system actually works before committing full resources.

**Purified into:**
- Offer validation (does this solve a real problem for a real person willing to exchange value?)
- Positioning validation (does this message land with the right person at the right moment?)
- Message validation (is the language precise, distinct, and resonant?)
- Channel validation (does this distribution path reach the intended audience efficiently?)
- Pricing validation (does this price reflect the perceived value accurately?)
- Prototype validation (does a minimum viable version of this prove the core hypothesis?)
- Market testing (traction signals vs vanity metrics)
- Traction hypothesis testing (if X is true, we will observe Y within Z time)
- Comparative benchmark testing (how does this perform against the category leader?)
- Simulation of execution paths (what breaks if conditions change?)

**Core law:** Validation before commitment is the difference between calculated risk and gambling.

**Anti-pattern:** Building to completion before testing the core hypothesis — spending 90% of resources to discover what a 1% test would have revealed.

**Benchmark standard:** A validation system should answer its core question with minimum resource expenditure. If testing costs more than a partial build, the test design is wrong.
